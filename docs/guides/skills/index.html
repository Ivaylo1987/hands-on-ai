Chapter 01 — Understand

The big picture

Before you build anything, you need to understand what you're building and why it works. This chapter gives you the mental model.

📚
What you'll learn: What a skill is, the folder structure (SKILL.md + optional scripts/references/assets), how the 3-level progressive disclosure system works, and the three design principles that make skills composable and portable.

Skills are powerful when you have repeatable workflows: generating frontend designs from specs, conducting research with consistent methodology, creating documents that follow your team's style guide, or orchestrating multi-step processes.

The skill folder

Click each file to learn what it does.

my-awesome-skill/
📄SKILL.mdRequired
📁scripts/Optional
📁references/Optional
📁assets/Optional
SKILL.md — YAML frontmatter + Markdown instructions. The only required file. Must be exactly SKILL.md (case-sensitive). No variations accepted.

Three design principles

Progressive disclosure

Three-level loading: metadata always loaded (~100 words), body on trigger (<500 lines), resources on demand (unlimited). Minimizes token usage.

Composability

Claude loads multiple skills simultaneously. Your skill should work alongside others — don't assume it's the only one.

Portability

Works identically across Claude.ai, Claude Code, and API. Create once, works everywhere. The open standard lives at agentskills.io.

How progressive disclosure works

Click each level to see how Claude loads your skill in stages.

1
Metadata (YAML frontmatter)
Always loaded • ~100 words
Always in Claude's system prompt. Just the name and description — how Claude decides if your skill is relevant. Keep descriptions under 1024 characters but specific and actionable.
2
SKILL.md body
On trigger • <500 lines ideal
Loaded when Claude thinks the skill is relevant. Contains your full instructions, workflow steps, examples, and error handling. If it's getting long, move details to reference files.
3
Linked files
On demand • Unlimited
Scripts, references, assets in subdirectories. Claude loads only what's needed. Scripts execute without being read into context — no token cost for running code.
Chapter 02 — Plan

Plan your skill

You know what a skill is. Now decide what yours should do. Clear planning prevents wasted effort — most skill problems trace back to vague use cases.

↪ Building on Chapter 1: you understand the folder structure and progressive disclosure. Now you'll decide what to put inside.
📚
What you'll learn: How to define concrete use cases (trigger, steps, result), the three categories of skills (document creation, workflow automation, MCP enhancement), and how to set measurable success criteria.

Define your use cases

Start with 2-3 concrete scenarios. Ask: What does a user want to accomplish? What steps are needed? Which tools (built-in or MCP)?

Use case template
Use Case: Project Sprint Planning
Trigger:  User says "help me plan this sprint" or "create sprint tasks"
Steps:
  1. Fetch current project status from Linear (via MCP)
  2. Analyze team velocity and capacity
  3. Suggest task prioritization
  4. Create tasks in Linear with proper labels and estimates
Result: Fully planned sprint with tasks created

Three categories of skills

Category 1

Document & asset creation

Consistent output: documents, presentations, code, designs. Uses Claude's built-in capabilities — no external tools required.

Example: frontend-design skill — "Create distinctive, production-grade frontend interfaces."
Category 2

Workflow automation

Multi-step processes with consistent methodology, templates, validation gates, and iterative refinement loops.

Example: skill-creator skill — "Interactive guide for creating new skills."
Category 3

MCP enhancement

Workflow guidance on top of MCP tool access. Coordinates multiple MCP calls and embeds domain expertise.

Example: sentry-code-review — "Analyzes and fixes bugs using Sentry data via MCP."

Success criteria

Set targets up front — rough benchmarks rather than precise thresholds. Aim for rigor but accept an element of vibes-based assessment.

Quantitative

90% trigger rate — test 10-20 queries
Fewer tool calls — compare with/without
0 failed API calls — monitor MCP logs

Qualitative

No next-step prompting — user doesn't redirect
No corrections needed — run 3-5 times
First-try success — new user completes task
Chapter 03 — Structure

Get the structure right

You have use cases and success criteria. Now create the actual folder and YAML. Get the structure wrong and Claude will never find your skill.

↪ Building on Chapter 2: your use cases tell you the skill name, and your trigger phrases go straight into the description field.
📚
What you'll learn: Folder naming conventions, YAML frontmatter format (required and optional fields), the 64-character name limit, security restrictions, and common upload errors to avoid.

Naming rules

✓ Correct
Folder: notion-project-setup (kebab-case)
File: SKILL.md (exact casing, no variations)
Name field: my-cool-skill (max 64 chars)
No README.md inside the skill folder
✗ Wrong
Folder: Notion Project Setup (spaces)
File: skill.md / SKILL.MD
Name: My Cool Skill / notion_project_setup
⚠
Security: No XML brackets (< >) in frontmatter. No "claude" or "anthropic" in names (reserved). Frontmatter appears in the system prompt — malicious content could inject instructions.

YAML frontmatter

This is how Claude decides whether to load your skill. The minimal version is just two fields:

Minimal — all you need to start
---
name: your-skill-name
description: What it does. Use when user asks to [specific phrases].
---

Full reference with all optional fields

YAML — Complete
---
name: skill-name                       # required, kebab-case, max 64 chars
description: [required]                 # required, max 1024 chars
license: MIT                             # optional (MIT, Apache-2.0)
allowed-tools: "Bash(python:*) WebFetch" # optional (Claude Code CLI only)
compatibility: Claude Code              # optional, 1-500 chars
metadata:                                # optional: custom key-value pairs
  author: Company Name
  version: 1.0.0
  mcp-server: server-name
  tags: [project-management, automation]
---
Chapter 04 — Write

Write your skill

Your folder exists with valid YAML. Now fill it with instructions Claude will actually follow. The quality of your writing directly determines how well the skill performs.

↪ Building on Chapter 3: your YAML frontmatter is set. Now write the description that triggers correctly and the body that guides Claude through your workflow.
📚
What you'll learn: How to write descriptions that trigger correctly (WHAT + WHEN + trigger phrases, always in third person), how to structure the SKILL.md body with the recommended template, and three core writing principles from Anthropic's official best practices.

Three writing principles

Concise is key

The context window is a public good. Challenge every paragraph: "Does this justify its token cost?" Only add what Claude doesn't already know.

Match the degrees of freedom

High: General guidance (code reviews). Medium: Templates with parameters (reports). Low: Exact scripts for fragile operations (migrations).

Explain the why

Claude is smart. Explain reasoning instead of rigid rules — the model goes beyond rote instructions when it understands purpose.

The description field

📝
Write in third person. "Processes Excel files and generates reports" — not "I can help you." The description is injected into the system prompt; inconsistent POV causes discovery problems.
✓ Good descriptions
Specific: "Analyzes Figma design files and generates developer handoff documentation. Use when user uploads .fig files or asks for design specs."

With triggers: "Manages Linear project workflows. Use when user mentions sprint, Linear tasks, or create tickets."
✗ Bad descriptions
Too vague: "Helps with projects."

No triggers: "Creates sophisticated multi-page documentation systems."

Too technical: "Implements the Project entity model with hierarchical relationships."

SKILL.md template

Recommended structure
---
name: your-skill
description: [What + When + Triggers]
---

# Your Skill Name

# Step 1: [First Major Step]
Clear explanation of what happens.
```bash
python scripts/fetch_data.py --project-id PROJECT_ID
```
Expected output: [describe what success looks like]

# Examples
User says: "Set up a new marketing campaign"
Result: Campaign created with confirmation link

# Troubleshooting
## Error: [Common error message]
Cause: [Why it happens]
Solution: [How to fix]
🛠
Bundle scripts for critical checks. Code is deterministic; language isn't. For operations where reliability matters, use a script rather than hoping instructions are followed exactly.
🔗
Keep references one level deep. SKILL.md links to reference files directly. Don't chain: SKILL.md → advanced.md → details.md. Claude may only partially read nested files.
Chapter 05 — Patterns

Build with proven patterns

You can write a skill from scratch. But you don't have to start from zero — these battle-tested patterns give you a head start for the most common use cases.

↪ Building on Chapter 4: you know how to write instructions. Now pick the pattern that matches your use case from Chapter 2 and adapt its template.
📚
What you'll learn: The problem-first vs. tool-first framing (Home Depot analogy), how MCP and skills work together ("kitchen analogy"), and five proven patterns with full code examples you can copy and adapt.
🏠
Problem-first vs. tool-first: Think of it like Home Depot. Walk in with a problem ("fix a cabinet") and an employee points to the right tools. Or pick out a drill and ask how to use it. Problem-first = user describes outcomes, skill handles tools. Tool-first = user has MCP access, skill provides expertise.

MCP + Skills: the kitchen analogy

MCP provides the professional kitchen (tools, ingredients, equipment). Skills provide the recipes (step-by-step instructions). Together, they enable complex tasks without users figuring out every step.

MCP (Connectivity)Skills (Knowledge)
Connects Claude to your serviceTeaches Claude how to use it effectively
Real-time data access & tool invocationWorkflows and best practices
What Claude can doHow Claude should do it

Five patterns

Click any pattern to see the full code example.

1Sequential workflow orchestration▼

Use when: Multi-step processes in a specific order.

Onboard new customer
# Step 1: Create Account
Call MCP tool: `create_customer`
Parameters: name, email, company

# Step 2: Setup Payment
Call MCP tool: `setup_payment_method`
Wait for: payment method verification

# Step 3: Create Subscription
Call MCP tool: `create_subscription`
Parameters: plan_id, customer_id (from Step 1)

# Step 4: Send Welcome Email
Call MCP tool: `send_email`
Template: welcome_email_template

Key: Explicit step ordering, dependencies, validation at each stage, rollback for failures.

2Multi-MCP coordination▼

Use when: Workflows span multiple services.

Design-to-dev handoff
# Phase 1: Design Export (Figma MCP)
1. Export design assets, generate specs, create manifest

# Phase 2: Asset Storage (Drive MCP)
1. Create project folder, upload assets, generate links

# Phase 3: Task Creation (Linear MCP)
1. Create dev tasks, attach assets, assign team

# Phase 4: Notification (Slack MCP)
1. Post handoff summary to #engineering

Key: Phase separation, data passing between MCPs, validation before advancing.

3Iterative refinement▼

Use when: Output quality improves with iteration (reports, reviews, content).

Report generation
# Draft → Fetch data, generate first draft, save temp file
# Check → Run `scripts/check_report.py`, identify issues
# Refine → Fix issues, regenerate sections, re-validate
# Ship → Apply formatting, generate summary, save final

Key: Explicit quality criteria, validation scripts, know when to stop.

4Context-aware tool selection▼

Use when: Same outcome, different tools depending on context.

Smart file storage
# Decision Tree
Large files (>10MB) → cloud storage MCP
Collaborative docs → Notion/Docs MCP
Code files → GitHub MCP
Temporary → local storage

# Then: Call tool, apply metadata, generate link, explain choice

Key: Clear decision criteria, fallbacks, transparency about choices.

5Domain-specific intelligence▼

Use when: Your skill adds specialized knowledge beyond tool access.

Payment compliance
# Compliance Check
Fetch transaction → check sanctions → verify jurisdiction → assess risk

# Process
IF passed: process payment with fraud checks
ELSE: flag for review, create compliance case

# Audit
Log checks, record decisions, generate report

Key: Domain expertise in logic, compliance before action, audit trail.

Chapter 06 — Test

Test & refine

You have a working skill. But "works once" isn't enough — you need to know it triggers reliably, produces correct output, and actually outperforms not having a skill at all.

↪ Building on Chapter 5: your skill is written using a proven pattern. Now validate it against the success criteria you defined in Chapter 2.
📚
What you'll learn: Three areas to test (triggering, functional, performance), the "Claude A / Claude B" development pattern, how to use the skill-creator for rapid iteration, and how to read under/over-triggering signals.
💡
Pro tip: Iterate on a single challenging task until Claude succeeds, then extract the winning approach into a skill. Faster signal than broad testing.

Three areas of testing

1. Triggering — does it load when it should?

How to test: Write 3-5 queries that should trigger your skill and 3-5 that shouldn't. Send each to Claude with the skill installed. Check whether it actually loaded the skill or ignored it. If you're unsure, ask Claude directly: "When would you use the [skill-name] skill?" — Claude will quote the description back, revealing what it sees.

Example: testing a "project-setup" skill
Should trigger (send these and verify skill loads):
  "Help me set up a new ProjectHub workspace"
  "Initialize a ProjectHub project for Q4 planning"

Should NOT trigger (verify skill stays dormant):
  "What's the weather in San Francisco?"
  "Help me write a Python function"

Debug if unclear:
  "When would you use the project-setup skill?"
  → Claude quotes description back → adjust if gaps found

2. Functional — does it produce the right output?

How to test: Pick a realistic task your skill should handle. Run it 3-5 times with the skill enabled. Check: Did it follow the steps in order? Did all API/MCP calls succeed? Does the output match what you'd expect? Run the same task without the skill to compare — the skill version should be noticeably better or faster.

Example: testing a sprint planning skill
Prompt:  "Plan next sprint for Project Alpha with 5 tasks"

Check each run for:
  ✓ Project fetched from Linear (Step 1 followed)
  ✓ Velocity calculated from last 2 sprints (Step 2)
  ✓ 5 tasks created with labels and estimates (Step 3)
  ✓ No API errors in the process
  ✓ Output is consistent across 3 runs

Compare against no-skill baseline:
  Without skill → user had to explain each step manually
  With skill → automatic workflow, same quality every time

3. Performance — is it measurably better?

How to test: Run the same task with and without the skill. Count three things: messages exchanged, failed API calls, and total tokens consumed. The skill version should require fewer messages, zero failures, and roughly half the tokens.

Without skill (baseline)
User explains workflow manually each time
15 back-and-forth messages
3 failed API calls requiring retry
12,000 tokens consumed
With skill
Automatic workflow execution
2 clarifying questions only
0 failed API calls
6,000 tokens consumed

"Claude A / Claude B" development pattern

The officially recommended approach: use two Claude instances working together.

Claude A — the designer

Work through the task using normal prompting. Notice what context you repeatedly provide. Ask Claude A to capture the pattern into a SKILL.md.

Claude B — the tester

Open a separate session with the skill loaded. Give it real tasks. Observe where it struggles. Bring observations back to Claude A for improvements.

Iteration signals

Undertriggering

Signs: Doesn't load when it should
Fix: Add keywords, technical terms, trigger phrases to description

Overtriggering

Signs: Loads for unrelated queries
Fix: First, make the description more specific. If that's not enough, add "Do NOT use for..." to explicitly exclude the conflicting cases.

Execution issues

Signs: Inconsistent results, failures
Fix: Improve instructions, add error handling, bundle scripts

Chapter 07 — Ship

Ship & share

Your skill is tested and working. Now get it to your team, handle the issues that come up in the real world, and make sure people actually adopt it.

↪ Building on Chapter 6: testing confirmed the skill works. Now upload it, share it with your team, and iterate on the feedback that comes in from real usage.
📚
What you'll learn: How to upload skills (claude.ai vs. API vs. Claude Code), the key difference between per-user and org-wide distribution, how to share with your team, how to troubleshoot the six most common issues, and how to position your skill for adoption.
⚠
Security: Use Skills only from trusted sources. Malicious Skills can direct Claude to invoke tools in unexpected ways. Audit all files (SKILL.md, scripts, resources) from unknown sources.

Upload your skill

Claude.ai

Settings > Features > Upload skill (zip). Pro, Max, Team, Enterprise plans. Custom skills are per-user — not shared org-wide.

Claude Code

Place in .claude/skills/ directory. Auto-discovered, no upload needed.

API

/v1/skills endpoint. Up to 8 skills per request via container.skills. API skills are shared org-wide. Requires 3 beta headers.

Agent SDK

Skills in .claude/skills/ with allowed_tools: ["Skill"]. Auto-discovered.

Share with your team

The right sharing method depends on how your team uses Claude. Here are the options, from simplest to most controlled:

Option 1: Git repo (recommended for most teams)

How: Commit your skill folder to a shared repo (e.g. your-org/claude-skills). Each team member clones it and either uploads the zip to Claude.ai or places it in their .claude/skills/ for Claude Code.

Best for: Teams using Claude Code. Skills auto-discover from the filesystem, and Git gives you version history, code review via PRs, and easy rollback.

Example structure:

Repo layout
your-org/claude-skills/
├── sprint-planner/
│   └── SKILL.md
├── code-review/
│   ├── SKILL.md
│   └── scripts/validate.py
└── README.md          # repo-level only, NOT inside skill folders

Option 2: API upload (org-wide, centrally managed)

How: Upload via the /v1/skills API endpoint. Skills become available to all workspace members automatically — no individual setup needed.

Best for: Teams using the API or Agent SDK for production deployments. Supports versioning and pinning.

Key detail: This is the only method that shares skills org-wide. Claude.ai uploads are always per-user.

Option 3: Plugin marketplace (Claude Code)

How: Package skills as a Claude Code plugin. Team members install with one command: /plugin install your-plugin. Skills from plugins are auto-namespaced to avoid conflicts.

Best for: Distributing multiple related skills as a bundle.

Option 4: Zip + manual share

How: Zip the skill folder and share via Slack, email, or a shared drive. Each person uploads it in Claude.ai via Settings > Features.

Best for: Quick sharing with 1-2 people. Doesn't scale well — no version control, each person manages their own copy.

💡
Which to pick: If your team uses Claude Code → Git repo (skills auto-load from the filesystem). If you need org-wide control → API upload (the only method that shares automatically). If you're just sharing with a colleague → zip it and send.

Troubleshooting

Click any card to reveal the fix.

Skill won't upload

Naming or YAML errors
"Could not find SKILL.md": Must be exactly SKILL.md. Verify: ls -la.

"Invalid frontmatter": Missing --- delimiters or unclosed quotes.

"Invalid name": Use my-cool-skill, not My Cool Skill.

Doesn't trigger

Skill never loads automatically
Is description too generic? Has trigger phrases? Mentions file types?

Debug: Ask Claude: "When would you use the [skill] skill?" Adjust based on response.

Triggers too often

Loads for unrelated queries
Add negative triggers: "Do NOT use for simple exploration."
Be more specific: Processes documents → Processes PDF legal documents for contract review

MCP calls fail

Skill loads but tools error
1. Check MCP connection in Settings
2. Verify API keys and scopes
3. Test MCP without the skill first
4. Verify tool names (case-sensitive)

Instructions ignored

Claude doesn't follow them
Put critical instructions at the top. Replace vague language with specific checks. Bundle validation scripts for reliability. Add "Take your time, quality > speed" to user prompts.

Slow responses

Large context degradation
Keep SKILL.md under 5,000 words. Move detail to references/. Reduce active skills if 20-50+ enabled.
